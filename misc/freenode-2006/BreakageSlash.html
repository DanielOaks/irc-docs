[edit]

Intent

Disambiguate channel names with respect to registered group names and their projects. Currently the use of the characters '.' and '-' can create confusion, since '.' is used to preface both group cloak and DNS TLD's, and '-' is used to separate group names in channel names from channel description material, as in: 

 #foo
 #foo-es
 #foo-devel

etc.


[edit]

Summary of Changes

1. The following formats would be used for group-registered channels: 

 #group
 #group/project-or-subgroup

2. Separate #project-or-subgroup channels might be kept or deleted.

3. Cloak hierarchies would be left-major, rather than the right-major hierarchy employed with DNS.

4. General format: 

 group/inside-material/uniqueid

rather than the current 

 uniqueid.inside-material.group-or-project

We might want to go to a more formal cloak specification, such as 

 group/[project/][role/]username

where username would be a unique freenode user name.


[edit]

Status

Half-baked.


[edit]

Caveats

1. At some point, group cloaks may not even be needed due to having affiliations listed in WHOIS and/or in FreenodeRegistry.

2. Various ideas are being bandied about in connection with the essential breakage and they should be addressed separately.


[edit]

Server

1. Possible semantic ambiguity between user names and cloaks containing neither "/" nor ".". Make sure user cloaks contain a minimum of one of those two characters.

2. Channel moves may require extensive use of ChannelForwarding, which is known to be broken in irssi and possible others. This needs fixing, probably at the server level.


[edit]

Client

1. In at least one major graphical client (mIRC? xchat?), channels with special characters in their names won't appear on LIST windows unless a specific configuration option is active. Does this apply to '/'?

2. Are there clients which will not accept '/' in a +Iebq hostmask?

3. The '/' has been used in some ircd variants in channel-ban-by-netmask; are there clients or scripts which will be confused by multiple '/' characters or by a '/' in an inappropriate place for a netmask?

4. Will specifying a hostname without a '.' and with a wildcard character break some client or banning script behavior?

5. For clients which accept irc: URI's: are additional '/' characters after the beginning of a channel name just interpreted as channel name characters? If not, we may have problems. Research.

6. Channel moves may require extensive use of ChannelForwarding, which is known to be broken in irssi and possible others. See Server issues.


[edit]

User Assumptions

1. If we have both right-major DNS hostnames with '.' and left-major cloaks with '/', will various common generic bans break? Generic bans are observed to be most common behavior for TraditionalIRCUsers. See Social Fabric issues.

2. Cloaks ending with /[0-9.]* look like ranges (e.g. something/23, something/255.255.254.0) so should not be used

[edit]

User Convenience

1. Channel names could be longer, so typing in the channel name on a join could be less convenient. If channel joins can be done in freenodeRegistry or online, via a StaticJoin model, that might mitigate the problem.


[edit]

Social Fabric

1. Left-major hostnames, with '/' instead of '.', will annoy TraditionalIRCUsers.

2. Any problems with banning scripts may annoy their users, which will be predominantly TraditionalIRCUsers.

3. Major channel naming changes may inconvenience existing groups. Possible workarounds include ChannelForwarding, which will need cleaning up (see Server and Client issues).
