What Atheme+ is/will be

Atheme+ is an exciting new service that will be offered by atheme.org's IRC activity for users of the Atheme IRC platform. This document is intended to explain what Atheme+ is, what it will eventually do, as well as attempt to dispel myths about it.

Current problems with IRC identity management

IRC, including public-facing networks running the Atheme IRC platform, is generally fragmented and lacks infrastructure for maintaining identity across multiple networks. Such a lack of centralised identity-verification services causes a significant problem for extending IRC with semantic information about users people are interacting with via IRC.

Atheme+ solves this issue by providing cross-network identity validation and management for all individuals using Atheme-based networks. It accomplishes this by:

  • taking advantage of new Atheme 7 features such as unique entity identifiers (UEIDs) and mapping them to Atheme+ network entities:
    • consider two networks, StaticBox and freenode.
    • on StaticBox, your account's UEID is AAY781EZQ.
    • on freenode, your account's UEID is HGYAEZ87D.
    • Atheme+ can correlate both of these accounts to a known Atheme+ account and denote that data in NickServ INFO output:
      -NickServ- Atheme+ ID: nenolod@dereferenced.org
  • providing synchronization of account settings, ensuring a consistent user experience across all networks linked to an Atheme+ ID
  • providing synchronization of channel settings, ensuring a consistent channel ACL across all networks linked to an Atheme+ ID:
    • like in our previous example, consider two networks, StaticBox and freenode.
    • on StaticBox, you have founder-level access on #atheme as Atheme+ mapped UEID: StaticBox:AAY781EZQ.
    • on freenode, you have founder-level access on #atheme as Atheme+ mapped UEID: freenode:HGYAEZ87D.
    • now consider another account registered to both networks, as StaticBox:YZN2AENCZ and freenode:XEZAN71E5.
    • because you have founder-level access to #atheme on both networks, Atheme+ will give channel access to both StaticBox:YZN2AENCZ and freenode:XEZAN71E5 when you change flags on either network because Atheme+ knows that it is the same person.

Current problems with IRC and the web

While Atheme's XMLRPC API server has enabled the creation of a large ecosystem of applications designed to integrate with Atheme, (such as Iris, Atheme-Web, phpAtheme and EGS,) many of the applications are cumbersome to configure or provide an incomplete user experience.

Atheme+ solves this issue by providing a consistent UI for managing all accounts linked to an Atheme+ identity, which allows for plug-and-play web integration.

Current problems with discovering IRC content

While IRC search engines, such as SearchIRC and IRCDriven provide a fairly decent way of discovering content, the actual experience is rather poor, and depends on indexing bots.

Atheme+ solves this issue through an optional statistics-push module, which provides channel lists and other general information about a network, updated in near-realtime. Additionally, the statistics-push module can provide information like URL to a qwebirc/iris instance for optimal integration of new users into the network.

Dispeling myths

  • Atheme+ is not OPT-IN: in reality, you don't have to use Atheme+ at all, networks do not have to use it either, and if networks/users don't want to use Atheme+, then things continue as they always have.
  • Atheme+ is a security risk: honestly, not really. the people who designed Atheme, and Atheme+ have considerable experience in the information security field. using Atheme+ is no more risky than using the XMLRPC server.
  • Atheme+ will allow spammers to register resources on my network without using them: Atheme+ doesn't have any facility for pre-registering a channel.
  • Atheme+ will have an Anope 1.9 module: this may or may not happen depending on how successful it is on Atheme networks. Integration would be somewhat limited with Anope networks, but we will see what we can do.
